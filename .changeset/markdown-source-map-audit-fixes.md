Hardens the markdown source-map so `resolveSelectionByPath` can never silently report invalid line ranges. Each fix is locked in by a slow-suite regression test.

- `resolveSelectionByPath` now returns a `resolved: boolean` discriminator. When neither endpoint maps to any segment, the result is `{ resolved: false, startLine: 1, endLine: 1, extractedMarkdown: '' }` so callers can refuse to build line references instead of silently falling back to line 1.
- `attachDescendantSources` now matches mdast children to Slate children by path tag instead of array position. `buildMdastNode` (and the list-paragraph constructor in `listToMdastTree`) propagate the originating Slate path onto the produced mdast node via `__sourceMapSlatePath`, so custom MDX serializers that inject, drop, or reorder mdast children no longer shift the source-map zipper and assign each Slate child the line range of a different mdast node. `attachListSources` uses the same tag for list paragraphs, with a positional fallback for callers that don't tag.
- `slateNodeKind` no longer emits a generic `'block'` segment for inline elements (links, mentions, dates, etc.) inside MDX containers — it consults `editor.api.isInline` and returns `null` for them. Inline elements occupy zero lines and confused the line-range math.
- `comparePoints` correctly orders mixed-depth selection points. The previous zero-padding compared an element-level child-index against a leaf-level character-offset, which silently flipped the order for selections like `{ path: [3], offset: 2 }` vs `{ path: [3, 1], offset: 5 }`. The element-level offset is now expanded into the same coordinate space as the deeper path, and the function is exported for direct testing.
- `findEndSegment` now picks the descendant segment with the largest `endLine` instead of whichever happened to be iterated last. The previous behaviour only worked because `allSegments` is incidentally sorted by path today; any future caller passing an unsorted array would have got the wrong range end.
- `pushSegment` strips a trailing newline from a handler's emitted markdown before counting line breaks. Standard mdast-util-to-markdown block handlers don't emit trailing newlines, but custom serializers (particularly MDX wrappers) sometimes do — without this defence a trailing `\n` pushed `endLine` one row past the actual content onto a blank line.
